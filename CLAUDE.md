## 🧠 記憶の引き継ぎ機能（セッション記録）

Claude Code は毎回新しいセッションで起動されるため、前回の会話内容を覚えていません。この問題を解決するため、**セッション記録**を使って「記憶の引き継ぎ」を実現します。

### 仕組み

1. **`dr-s`（記録開始）**: 前回のセッション記録を読み込んで振り返りを表示
2. **日々の習慣記録と学習**: 通常通り習慣と学習を記録
3. **`dr-f`（記録完了）**: 今日のセッション内容を `memory/sessions/` に保存

### セッション記録に含まれる情報

- **達成した習慣**: 今日完了した習慣と未達成の習慣
- **今日の気づき・メモ**: 学習内容や印象的だったこと
- **学習の進捗**: SQL ドリルと Writebook の進捗
- **セッションの特徴**: 使用したスキルや特徴的な出来事
- **次回へのメモ**: 未達成の習慣や次回やりたいこと

### メリット

- ✅ **継続性の可視化**: 毎日の習慣がどれくらい続いているか把握できる
- ✅ **前回の振り返り**: 前回のセッションで何をしたか、何を気づいたかを思い出せる
- ✅ **次回へのリマインダー**: 前回の「次回へのメモ」を自動的に表示
- ✅ **学習の進捗管理**: どこまで学習が進んだかを追跡

### 例: セッション記録のフォーマット

```markdown
# 2026-04-02 セッション記録
```

## Claude Code の処理フロー

### ⭐ 1 日の初めの作業フロー（最優先）

ユーザーから「今日の記録を開始」「おはよう」などの声かけがあり、**その日の初めてのやり取り**の場合：

#### ステップ 1: 日付の確認

- システムの日付（`<env>` の Today's date）を確認
- 例: 2026-04-11

#### ステップ 2: 日次ファイルの存在確認

- `daily/2026-04-11.md` が存在するか確認
- **存在する場合**: そのファイルを開いて更新モードで作業
- **存在しない場合**: 次のステップへ進む（新規作成フロー）

#### ステップ 3: 新規作成フローの開始

日次ファイルが存在しない場合、以下の手順で作業を開始：

1. **ブランチの作成**

   ```bash
   git checkout -b 2026-04-11
   ```

   - ブランチ名は日付形式 `YYYY-MM-DD`
   - main ブランチから分岐

2. **日次ファイルの作成**

   - `daily/TEMPLATE.md` を読み込む
   - `daily/2026-04-11.md` として新規作成
   - 日付部分（`# YYYY-MM-DD`）を実際の日付に置換

3. **ユーザーへの確認**
   - 「2026-04-11 の記録を作成しました！」
   - 完了した習慣があれば報告を促す

#### ステップ 4: 習慣の記録

- ユーザーから報告された完了済み習慣をチェック済みにする
- 回数の記録が必要な項目（筋トレなど）は具体的な回数を記載

#### 重要な判断基準

- **「その日の初めてのやり取り」の判定**:
  - 会話の最初の方で記録開始を依頼された場合
  - まだ日次ファイルが存在しない場合
- **エラーハンドリング**:
  - ブランチ作成に失敗 → ユーザーに状況を報告
  - 日付の不一致 → システム日付を優先し、ユーザーに確認

### 2. daily/ への記録（通常の記録更新）

日次ファイルが既に存在し、追加の習慣や学習内容を記録する場合：

#### 基本習慣の更新

- ユーザーから報告された習慣のチェックボックスを `[x]` に変更
- 回数がある項目は実際の回数を記載
  - 例: `- [x] 腕立て伏せ 15 回完了`
- 目標に満たない場合でもそのまま記録（正確性を優先）

#### SQL ドリルの記録

ユーザーから「SQL ドリル P.45-50 完了」のような報告があった場合：

1. **daily/ の更新**

   ```markdown
   ### SQL ドリル

   - [x] 今日の進捗: P.45-50、SELECT文の練習
   - メモ: WHERE句の条件指定が理解できた
   ```

2. **詳細記録は不要**
   - SQL ドリルは物理本 + 手書きメインのため、daily のメモのみでOK
   - 特別な気づきがあれば「今日の振り返り」セクションに記載

#### Writebook コードリーディングの記録

ユーザーから「Writebook の投稿機能を読んだ」のような報告があった場合：

1. **daily/ の更新**

   ```markdown
   ### Writebook コードリーディング

   - [x] 今日読んだ部分: app/controllers/posts_controller.rb, app/models/post.rb
   - メモ: Hotwire Turbo の実装パターンが参考になった
   - 詳細ノート: （詳細を残したい場合のみリンクを追加）
   ```

2. **詳細ノートの作成（任意）**

   - ユーザーが詳細を残したい場合のみ作成
   - **ユーザーに確認する**: 「詳細ノートを作成しますか？」
   - 作成する場合:
     - `learning/writebook/YYYY-MM-DD-{トピック}.md` を作成
     - daily からリンクを貼る
     - `learning/writebook/README.md` にもリンクを追加

### 3. Writebook 詳細ノートの作成（任意）

ユーザーが詳細ノートを残したい場合、以下の形式で作成：

#### ファイル名

```
learning/writebook/YYYY-MM-DD-{トピック}.md
```

例: `2026-04-11-post-creation.md`

#### 内容構成

```markdown
# {トピック名}

- 日付: 2026-04-11
- 読んだファイル: `app/controllers/posts_controller.rb`, `app/models/post.rb`
- daily 記録: [daily/2026-04-11.md](../../daily/2026-04-11.md)

## 理解したこと

### コントローラの実装
（コードや気づきを記録）

### モデルの設計
（バリデーション、関連、スコープなど）

## 気づき・学び

- ○○という実装パターンが参考になった
- △△の考え方が業務に活かせそう

## 関連ファイル

- `app/views/posts/_form.html.erb`
- `app/javascript/controllers/post_controller.js`
```

#### daily/ へのリンク追加

```markdown
- 詳細ノート: [learning/writebook/2026-04-11-post-creation.md](../learning/writebook/2026-04-11-post-creation.md)
```

#### README へのリンク追加

`learning/writebook/README.md` の「学習ノート一覧」セクションに追加：

```markdown
- 2026-04-11: [投稿機能の実装を読んだ](2026-04-11-post-creation.md) - [daily](../../daily/2026-04-11.md)
```

### 4. Git コミットと PR の作成

作業が完了したら、変更をコミットして PR を作成します。

#### コミットの作成

1. **変更内容の確認**

   ```bash
   git status
   git diff
   ```

2. **ステージングとコミット**

   ```bash
   git add daily/2026-04-11.md
   # Writebook の詳細ノートも作成した場合は追加
   git add learning/writebook/2026-04-11-*.md
   git add learning/writebook/README.md

   git commit -m "2026-04-11 の日次記録を追加"
   ```

3. **コミットメッセージのルール**
   - **必ず日本語**で記述
   - 簡潔に変更内容を要約
   - 例:
     - `2026-04-11 の日次記録を追加`
     - `SQL ドリル: P.45-50 完了`
     - `Writebook: 投稿機能の実装を読んだ`

#### PR の作成

1. **ブランチのプッシュ**

   ```bash
   git push -u origin 2026-04-11
   ```

2. **PR の作成**

   ```bash
   gh pr create --title "2026-04-11 の日次記録" --body "$(cat <<'EOF'
   ## 概要
   2026-04-11 の習慣記録と学習ノートを追加

   ## 変更内容
   - 基本習慣の記録
   - 学習内容の記録（SQL ドリル、Writebook）

   🤖 Generated with [Claude Code](https://claude.com/claude-code)
   EOF
   )"
   ```

3. **PR のルール**
   - タイトル: 日本語で簡潔に（例: `2026-04-11 の日次記録`）
   - Description: 日本語で詳細を記述
     - 概要セクション
     - 変更内容セクション
     - 必要に応じてテスト計画など
   - 最後に Claude Code の署名を追加

#### 重要な注意事項

- **すべて日本語で記述**（コミットメッセージ、PR タイトル、PR description）
- 1 日の作業が完了したらすぐに PR を作成
- PR はマージせずに残しておく（レビュー・記録用）
- **`git commit`、`git push`、PR 作成は、ユーザーから明示的に依頼があるまで実行しない**
  - ファイルの作成・編集まで自動で行うが、コミット以降はユーザーの指示を待つ
