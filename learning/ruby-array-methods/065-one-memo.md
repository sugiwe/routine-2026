# Ruby の `Array#one?` メソッドの早期リターン動作を検証した話

## きっかけ

Ruby の `Array#one?` メソッドについて学習ノートを作成した際、パフォーマンス特性について以下のように書いていました：

> `one?` は配列全体を走査します（早期リターンなし）
> 条件を満たす要素が見つかっても、2つ目がないか確認するため最後まで調べる

しかし、Gemini にレビューしてもらったところ、この記述が誤りだと指摘されました。

## Gemini の指摘内容

> one? のパフォーマンスに関する説明が不正確です。
>
> 実際には、one? は条件に一致する要素が2つ見つかった時点で false を返して評価を停止するため、早期リターンします。配列全体を走査するのは、条件に一致する要素が1つ（結果が true）または0個の場合のみです。

## 検証プロセス

### 1. 公式ドキュメントの確認

まず、Ruby の公式ドキュメントを確認しました：

- 英語版: https://docs.ruby-lang.org/en/4.0/Array.html#method-i-one-3F
- 日本語版: https://docs.ruby-lang.org/ja/latest/method/Array/i/one=3f.html

しかし、**どちらのドキュメントにも早期リターンの動作については明記されていませんでした**。

### 2. ソースコードの調査

Ruby の GitHub リポジトリで `enum.c` ファイルを探しましたが、WebFetch では実装の詳細まで確認できませんでした。

### 3. 実際の動作検証

ドキュメントでは判断できないため、**実際に Ruby コードで動作を検証**することにしました。

```ruby
# one? メソッドの早期リターンをテスト

puts "=== テスト1: 2つ目が見つかった時点で停止するか ==="
count = 0
result = [1, 2, 3, 4, 5].one? do |n|
  count += 1
  puts "  チェック中: #{n} (#{count}回目の呼び出し)"
  n > 2  # 3, 4, 5 が条件を満たす（3つ該当）
end
puts "結果: #{result}"
puts "ブロックが呼ばれた回数: #{count}回"
puts "期待: 早期リターンなら4回（3と4を見つけた時点で停止）、全走査なら5回"
```

#### 実行結果

```
=== テスト1: 2つ目が見つかった時点で停止するか ===
  チェック中: 1 (1回目の呼び出し)
  チェック中: 2 (2回目の呼び出し)
  チェック中: 3 (3回目の呼び出し)
  チェック中: 4 (4回目の呼び出し)
結果: false
ブロックが呼ばれた回数: 4回
```

**4回で停止しました！** 5つ目の要素（5）は評価されていません。これは早期リターンが実装されている証拠です。

### 4. 大規模配列での検証

さらに、大きな配列でも確認してみました：

```ruby
puts "=== 大きな配列（1000要素）での早期リターンテスト ==="
count = 0
result = (1..1000).to_a.one? do |n|
  count += 1
  n > 10  # 11以上の要素は990個（2つ目は12で見つかるはず）
end
puts "結果: #{result}"
puts "ブロックが呼ばれた回数: #{count}回"
```

#### 実行結果

```
結果: false
ブロックが呼ばれた回数: 12回
```

**1000要素の配列でも、わずか12回の評価で処理が完了**しました！これは、11番目の要素（値: 11）で1つ目が見つかり、12番目の要素（値: 12）で2つ目が見つかった時点で即座に `false` を返して停止したことを示しています。

### 5. 100万要素でさらに検証

最終確認として、100万要素の配列でも試してみました：

```ruby
count = 0
(1..1_000_000).to_a.one? do |n|
  count += 1
  n > 10
end
# 結果: false
# count は 12 のみ！
```

100万要素の配列に対しても、**たった12回の評価**で処理が完了しました。もし全走査していたら100万回の評価が必要だったはずです。

### 6. 全走査が必要なケースも確認

念のため、全走査が必要なケースも検証しました：

```ruby
# 条件を満たすのが1つだけの場合
count = 0
result = [1, 2, 3, 4, 5].one? do |n|
  count += 1
  n == 3  # 3だけが条件を満たす
end
# 結果: true
# count: 5回（全走査が必要）

# 条件を満たすのが0個の場合
count = 0
result = [1, 2, 3, 4, 5].one? do |n|
  count += 1
  n > 10  # どれも条件を満たさない
end
# 結果: false
# count: 5回（全走査が必要）
```

## 結論

**Gemini の指摘が正しかった**ことが実証されました。

### `Array#one?` の実際の動作

1. **2つ目が見つかった時点で即座に `false` を返す（早期リターン）**
   - 条件に一致する要素が2つ見つかると、残りの要素を評価せずに停止
   - 大きな配列でも、条件を満たす要素が前方にあればすぐに結果が出る

2. **配列全体を走査するのは以下の場合のみ**
   - 条件に一致する要素がちょうど1つの場合（結果が `true`）
   - 条件に一致する要素が0個の場合（結果が `false`）

### パフォーマンス特性

- **ベストケース**: O(2) — 先頭2つの要素が条件を満たす場合
- **ワーストケース**: O(n) — 条件を満たす要素が0個または1個の場合
- **平均ケース**: 条件を満たす要素が複数ある場合、早期リターンにより大幅に高速化

## 学んだこと

1. **公式ドキュメントにも書かれていない実装の詳細がある**
   - パフォーマンス特性は実装に依存するため、ドキュメントに明記されていないことが多い
   - 重要な動作は実際に検証することが大切

2. **AI レビューの価値**
   - Gemini が正確に早期リターンの実装を指摘してくれた
   - しかし、AI の指摘も鵜呑みにせず、実際に検証することが重要

3. **実証的アプローチの重要性**
   - ブロック呼び出し回数をカウントすることで、内部動作を明確に確認できた
   - 小さな配列から大きな配列まで段階的に検証することで、確信を持てた

4. **誤った前提で学習していたことに気づけた**
   - 「ちょうど1つかどうか」を確認するには全走査が必要だと思い込んでいた
   - 実際には「2つ以上あるか」を確認すれば十分なので、早期リターンが可能
