# モードレス（Modelessness）

- 日付: 2026-01-09
- URL: https://www.sociomedia.co.jp/9094

## メモ（daily より）

UI におけるモードとは、現在可能な操作を限定したり、操作手順を固定的に矯正する状態。モードがあると、ユーザーは作業の途中で考えを変えて別のことをする際にまずモードから抜けるという操作が必要になる。モードレスであればユーザーは自由な順序で作業を行うことができる。

## 深掘り・補足

### モードレスの定義

**モードレス**とは、「操作の意味が一定で、ユーザーが自由な順序で作業を行える状態」を指します。これは優れた UI デザインの基本条件の一つです。

### モードの問題点

モードがある UI では以下のような制約が生じます：

1. **操作の意味が状況依存**：同じボタンやキー操作でも、現在のモードによって異なる動作をする
2. **操作の制限**：現在のモードで許可された操作しか実行できない
3. **手順の強制**：特定の順序でしか作業を進められない
4. **作業フローの中断**：別の作業をしたい場合、まずモードから抜け出す必要がある

### 具体例：設定画面の比較

#### ❌ モーダルダイアログ（モードあり）

```
[設定] ボタンをクリック
  ↓
モーダルダイアログが表示される
  ↓
設定を変更
  ↓
[OK] または [キャンセル] を押さないと他の操作ができない
```

- 問題：設定中は他の画面を見たり操作したりできない
- ユーザーは「設定モード」に閉じ込められる

#### ✅ モードレスパネル（モードなし）

```
[設定] ボタンをクリック
  ↓
サイドパネルが表示される
  ↓
設定を変更しながら、メイン画面も同時に操作可能
  ↓
いつでも自由に他の作業に移れる
```

- 利点：設定とメイン作業を並行して進められる
- ユーザーの主導権が保たれる

### 実装のポイント

1. **ダイアログは最小限に**
   - 必須の確認や警告以外はモーダルダイアログを避ける
   - 設定や編集は可能な限りインラインまたはサイドパネルで

2. **いつでも戻れる設計**
   - ユーザーが作業を中断して別の操作をできるようにする
   - 「この操作を完了しないと次に進めない」状態を作らない

3. **操作の一貫性**
   - 同じボタンやキーは常に同じ意味を持つようにする
   - モードによって操作の意味が変わらないようにする

4. **UI をロックしない**
   - 処理中でも可能な操作は許可する
   - 長時間の処理はバックグラウンドで実行

### 関連する原則

- **ユーザーの主導権**：ユーザーがコントロールできる状態を保つ
- **オブジェクトベースの操作**：対象を選んでから操作を選ぶ方式
- **UI をロックしない**：処理中も他の作業を妨げない

### 例外：モードが必要な場合

完全にモードレスにできない場合もあります：

- **重要な確認**：削除や上書きなど、取り返しのつかない操作の前
- **排他的な操作**：同時に実行できない処理（例：ファイルの編集競合）
- **セキュリティ**：認証やアクセス制御が必要な場面

このような場合でも、モードの範囲を最小限にし、できるだけ早く通常状態に戻れるようにすることが重要です。
