# ユーザーの言葉を使う (Users' Language)

- 日付: 2026-01-11
- URL: https://www.sociomedia.co.jp/9108

## メモ（daily より）

UI で使う文言は、システム側の用語ではなくユーザーが普段使っている言い回しにする。これ結構やりがちなので注意したい。最近は UX ライティングという文脈で語られることも多い気がする。

## 深掘り・補足

### 基本概念

インターフェース上で使う文言は、システム内部の技術用語や運営者の業界用語ではなく、**ユーザーが普段使っている一般的な表現を用いる**ことが重要です。この原則は、ユーザーの認知的負荷を減らし、直感的に理解できる UI を実現するための基本です。

### 具体例：良い例と悪い例

#### フライト予約画面の例

**❌ 悪い例：業界用語を使用**
- 空港コードのみを表示：「NRT」「JFK」
- 一般ユーザーには理解が難しい
- 追加の認知的努力が必要

**✅ 良い例：ユーザーの言葉を使用**
- 一般的な空港名を表示：「東京（成田）」「ニューヨーク（ジョン・F・ケネディ）」
- 誰でも理解できる
- 必要に応じて補足情報（コード）を追加

#### その他の例

**❌ 悪い例**
- 「レコードを削除」→ データベース用語
- 「プロセスを終了」→ システム用語
- 「パラメータを設定」→ 技術用語

**✅ 良い例**
- 「記録を削除」「投稿を削除」
- 「作業を中止」「処理を止める」
- 「設定を変更」「オプションを選ぶ」

### UX ライティングとの関係

最近では「UX ライティング」という専門分野として確立されており、以下のような観点が重視されています：

1. **明確さ (Clarity)**：曖昧さを排除し、明確に伝える
2. **簡潔さ (Conciseness)**：必要最小限の言葉で表現する
3. **一貫性 (Consistency)**：同じ概念には同じ言葉を使う
4. **会話的 (Conversational)**：自然な話し言葉に近い表現を使う

### 関連する設計原則

- **メンタルモデル**：ユーザーの心的表現と一致させる
- **シグニファイア**：操作方法を明確に示す
- **マッピング**：操作と結果の対応関係を直感的にする

これらは全て、ユーザー中心の言語・表現を実現するための相互補完的な設計手法として機能します。

### 実践のためのチェックリスト

1. **ユーザー調査を実施**：ターゲットユーザーが実際に使っている言葉を調べる
2. **専門用語を避ける**：必要最小限に留め、説明を添える
3. **コンテキストを考慮**：使用場面に応じて適切な言葉を選ぶ
4. **ユーザビリティテスト**：実際のユーザーに理解できるか確認する
5. **継続的な改善**：フィードバックをもとに表現を洗練させる

### よくある落とし穴

- 社内用語をそのまま UI に使ってしまう
- 技術者目線での言葉選びになってしまう
- 業界標準だからという理由で専門用語を使ってしまう
- 翻訳アプリに頼りすぎて不自然な表現になる

「ユーザーの言葉を使う」ことは、単なる翻訳作業ではなく、**ユーザーの視点に立って考える**という UX デザインの根本的な姿勢を体現する原則です。
