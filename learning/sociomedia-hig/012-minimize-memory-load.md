# ユーザーの記憶に頼らない (Minimize the user's memory load)

- 日付: 2026-01-12
- URL: https://www.sociomedia.co.jp/9111

## メモ（daily より）

システムが表示するメッセージの内容やプロパティの値などについてユーザーが覚えていることを前提にしてはいけない。フォーム入力の不備があった際に、モーダルメッセージで注意を出すだけでなく、フォームの適切な場所に注意を表示していく、など。

## 深掘り・補足

### 基本原則

ユーザーが必要とする情報は、**それを参照する必要が生じたその時点で、すぐにアクセス可能であるべき**です。過去の情報を記憶しておくことに依存するインターフェース設計は避ける必要があります。

### 認知負荷の軽減

人間の短期記憶には限界があります（いわゆる「マジカルナンバー7±2」）。システム設計において、ユーザーに情報を記憶させることは以下の問題を引き起こします：

- **エラーの増加**: 記憶違いによる誤操作
- **効率の低下**: 情報を思い出すための時間のロス
- **ストレスの増加**: 「覚えていないといけない」というプレッシャー

### 良い例と悪い例

#### ❌ 悪い例：モーダルダイアログでのエラー表示

```
┌─────────────────────────────┐
│  エラーが発生しました        │
│  ・メールアドレスの形式が   │
│    正しくありません          │
│  ・パスワードは8文字以上    │
│    必要です                  │
│         [ OK ]               │
└─────────────────────────────┘

（ユーザーは「OK」を押した後、
  どの項目にエラーがあったか
  記憶しながらフォームに戻る）
```

**問題点**:
- エラー内容を記憶しないといけない
- エラー箇所を自分で探さないといけない
- 複数エラーがある場合、すべて覚えるのは困難

#### ✅ 良い例：インラインエラー表示

```
メールアドレス: [user@example]
                ⚠️ メールアドレスの形式が正しくありません

パスワード:     [••••••]
                ⚠️ パスワードは8文字以上必要です
```

**利点**:
- エラーが該当項目の近くに表示される
- 記憶する必要がない
- 修正しながら確認できる

### 実践的なガイドライン

1. **文脈内情報提供（Contextual Information）**
   - 必要な情報は、必要な場所に表示する
   - 例: ツールチップ、プレースホルダー、ヘルプテキスト

2. **認識 vs. 想起（Recognition vs. Recall）**
   - ユーザーに「思い出させる」のではなく「認識させる」
   - 例: コマンド入力ではなく、視覚的な選択肢を提供

3. **状態の可視化（Visibility of System Status）**
   - 現在の状態や選択肢を常に表示
   - 例: ステップインジケーター、現在地の表示

4. **一貫性のある配置（Consistent Placement）**
   - 同じ種類の情報は同じ場所に表示
   - 例: エラーメッセージは常に入力欄の下に表示

### 関連する原則

- **シグニファイア (Signifier)**: 操作可能な要素を明示する
- **一貫性 (Consistency)**: 同じ情報を同じ方法で表示する
- **ユーザーコントロール (User Control)**: いつでも情報にアクセスできる

### 実装上の注意点

- **Progressive Disclosure（段階的開示）**: 情報を一度に全て表示するのではなく、必要に応じて段階的に表示する
- **デフォルト値の活用**: 以前の入力内容を記憶・表示することで、再入力の手間を省く
- **視覚的な手がかり**: アイコン、色、位置などで情報を補強し、記憶に頼らない設計にする
