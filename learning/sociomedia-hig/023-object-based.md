## メモ（daily より）

要求からオブジェクトを抽出して UI に反映する。手続的な UI では、操作しづらく効率が悪くなっていく。

## 感想（daily より）

## 深掘り・補足

### オブジェクトベース UI（OOUI）とは

オブジェクトベース UI は、ユーザーが「**何をするか**」ではなく「**何に対して作用するか**」に焦点を当てた設計思想です。要求をそのまま手続き的になぞるのではなく、要求からオブジェクトを抽出して UI 設計の基準とします。

### 手続き的 UI との違い

**❌ 手続き的 UI の問題点：**
- 機能を線型に並べるだけになる
- 操作体系としてのまとまりに欠ける
- 効率が悪く分かりにくいシステムになる
- 例：「新規登録」「編集」「削除」を別々のページや機能として提示

**✅ オブジェクトベース UI の利点：**
- ユーザーがオブジェクトに直接働きかけながら目的を達成できる
- 操作の文脈が明確になる
- 直感的で効率的な操作が可能になる
- 例：レコード一覧を見せて、そこで各操作（編集・削除など）を実行できるようにする

### 具体例：本と著者の管理アプリケーション

**悪い例（手続き的）：**
```
- 本を新規登録
- 本を編集
- 本を削除
- 著者を新規登録
- 著者を編集
- 著者を削除
```
機能を列挙するだけで、操作の文脈が不明確。

**良い例（オブジェクトベース）：**
```
📚 本
  - 本の一覧（ここで追加・編集・削除が可能）
👤 著者
  - 著者の一覧（ここで追加・編集・削除が可能）
```
オブジェクト（本、著者）を中心に構成し、各オブジェクトに対して操作を行う。

### レコード管理システムの例

**悪い例：**
- 「新規レコード作成」画面
- 「レコード編集」画面
- 「レコード削除」画面

**良い例：**
- 「レコード一覧」画面
  - 各レコードに対して編集・削除が可能
  - 一覧の上部に「新規作成」ボタン

### 設計プロセス

GUI 設計には以下のサイクルが必要です：

1. **抽象的な要求** → ユーザーの目的を理解
2. **具象的な手続き** → 要求を実現する操作フローを導き出す
3. **抽象的な原理** → オブジェクトを抽出し、再び抽象化
4. **具象的なデザイン** → オブジェクト中心の UI を設計

### 注意点

- ごくシンプルなアプリケーション（機能が1〜2個程度）では、手続き的 UI でも十分使いやすい場合がある
- しかし、機能が増えていくにつれて、オブジェクトベース UI の利点が顕著になる
- 最初からオブジェクトベースで設計しておくことで、将来の拡張性が向上する

### 関連する原則

- **直接操作（Direct Manipulation）**：オブジェクトを直接操作できるようにする
- **見えるようにする（Visibility）**：オブジェクトとその状態を常に可視化する
- **一貫性（Consistency）**：同じオブジェクトには同じ操作体系を適用する
