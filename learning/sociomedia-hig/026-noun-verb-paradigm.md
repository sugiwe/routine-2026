# 名詞 → 動詞 の操作順序 (Noun-Verb Paradigm)

- 日付: 2026-01-26
- URL: https://www.sociomedia.co.jp/9275

## メモ（daily より）

対象物（名詞）を選んでからアクション（動詞）を選ぶようにする。「開く」や「編集」を押してからファイル一覧を表示するのではなく、ファイル一覧からファイルを選んだ上で「開く」「編集」などを選択できるようにする。

前に読んだ「オブジェクト指向 UI デザイン」のなかで一番納得感があり腑に落ちた話だ

## 深掘り・補足

### 基本原則

**Noun-Verb Paradigm（名詞→動詞パラダイム）** は、GUI デザインにおける基本的な操作順序の原則です。

- **推奨**: 対象物（名詞）を選択 → アクション（動詞）を実行
- **非推奨**: アクション（動詞）を選択 → 対象物（名詞）を選択

### なぜこの順序が重要なのか

#### 1. 操作の自由度

**良い例（名詞→動詞）:**
```
ファイル一覧 → ファイル A を選択 → 「開く」または「編集」または「削除」を選択可能
```
選択後に複数のアクションから選べるため、柔軟性が高い。

**悪い例（動詞→名詞）:**
```
「開く」ボタンを押す → モーダルが開く → ファイルを選択 → OK を押す
```
「開く」を選んだ時点で操作が固定され、他のアクション（編集・削除など）に切り替えられない。

#### 2. 操作待ちモードの問題

動詞→名詞の順序では「対象物の選択待ちモード」が発生します：

- ユーザーは別の操作ができなくなる
- モーダルダイアログで閉じ込められる
- 途中でキャンセルする手間が増える

#### 3. オブジェクト指向との整合性

プログラミングの世界では、オブジェクトがメッセージ（アクション）を受け取る構造が自然です：

```javascript
// オブジェクト指向的な考え方
file.open()    // ファイル（名詞）に対して開く（動詞）を実行
file.edit()    // 同じファイルに対して編集も可能
```

これは「名詞→動詞」の操作順序と一致します。

### 具体例

#### ✅ 良い例

**メールアプリ:**
1. メール一覧からメールを選択（名詞）
2. 「削除」「アーカイブ」「返信」などのボタンが有効になる（動詞）
3. 別のメールを選択すれば、同じボタンで別の対象に操作できる

**ファイル管理:**
1. Finder/Explorer でファイルを選択（名詞）
2. 右クリックメニューまたはツールバーで「開く」「名前変更」「削除」などを選択（動詞）

#### ❌ 悪い例

**古い Web フォーム:**
1. 「ファイルをアップロード」ボタンを押す（動詞が先）
2. ファイル選択ダイアログが開く（モーダル状態）
3. ファイルを選んで OK を押す（名詞が後）
4. 別のファイルをアップロードするには最初からやり直し

### 実践的なアドバイス

#### デザイン時のチェックポイント

1. **選択状態の明示**: 何が選ばれているかを視覚的に明確にする
2. **コンテキスト依存のアクション**: 選択されたオブジェクトに応じて利用可能なアクションを表示
3. **複数選択への対応**: 複数の名詞を選んでから一括でアクションを実行できるようにする

#### 例外的なケース

必ずしも名詞→動詞が適用できない場合もあります：

- **「新規作成」系のアクション**: 対象物がまだ存在しないため、動詞が先になる
- **グローバルな操作**: 特定のオブジェクトに依存しない操作（例: 「すべて更新」）

ただし、これらの場合でも、可能な限り名詞→動詞の原則に近づける工夫が求められます。

### オブジェクト指向 UI デザインとの関連

このパラダイムは **OOUI（Object-Oriented User Interface）** の中心的な概念です。OOUI では：

- UI はオブジェクト（名詞）を中心に構成される
- ユーザーはオブジェクトを直接操作する
- アクション（動詞）はオブジェクトのプロパティとして提供される

この考え方により、直感的で一貫性のある UI を実現できます。

### まとめ

- **名詞→動詞** の順序は操作の自由度を高める
- **動詞→名詞** はユーザーを操作待ちモードに閉じ込める
- オブジェクト指向プログラミングの原則と整合性がある
- 例外はあるが、基本原則として常に意識すべき
