# 32. 前提条件は先に提示する (Present Prerequisites First)

- 日付: 2026-02-01
- URL: https://www.sociomedia.co.jp/9318
- daily 記録: [daily/2026-02-01.md](../../daily/2026-02-01.md)

## ユーザーの第一印象（daily より）

事前に満たしておくべき前提条件を長い手続きの途中や最後になってから求めると、そこまでのユーザーの労力が無駄になる恐れがある。例えばアプリケーションの外部から参照する必要のある入力情報や、同意する必要のある規約など。

## 深掘り・補足

### 基本原則

ユーザーに時間と労力を投資させる前に、必要な前提条件をすべて明示することが重要です。長い手続きの途中や最後で「実は〇〇が必要でした」と伝えることは、ユーザーの信頼を損ない、離脱を招く原因となります。

### 典型的な悪い例

**複数ページにわたる入力フォームの最後で利用規約を提示**

```
ステップ 1: 個人情報入力 (5分)
  ↓
ステップ 2: 住所・連絡先入力 (3分)
  ↓
ステップ 3: 詳細情報入力 (5分)
  ↓
ステップ 4: 利用規約への同意 ← ここで初めて規約を見て同意できないと判断
  → それまでの13分の労力が無駄に
```

### 良い例

**最初に前提条件を明示**

```
ステップ 1: 利用規約の確認と同意 ← 先に提示
  ↓
ステップ 2: 個人情報入力
  ↓
ステップ 3: 住所・連絡先入力
  ↓
完了
```

### 前提条件として先に提示すべきもの

1. **同意が必要な規約**
   - 利用規約
   - プライバシーポリシー
   - 料金体系

2. **外部参照が必要な情報**
   - クレジットカード番号
   - マイナンバー
   - 保険証番号
   - パスポート番号

3. **事前準備が必要な条件**
   - 特定のソフトウェアのインストール
   - 推奨ブラウザの使用
   - 必要なファイル形式

4. **システム要件**
   - 対応OS・デバイス
   - 必要な権限（カメラ、位置情報など）
   - ネットワーク環境

### 実装のポイント

1. **チェックリスト形式で提示**
   - 「この手続きには以下が必要です」と明示
   - 各項目をチェックボックス化して準備を促す

2. **所要時間の明記**
   - 「約10分で完了します」など事前に伝える
   - 途中離脱しても再開できることを明示

3. **段階的な開示**
   - 最初に全体像を見せる
   - 各ステップの冒頭で必要なものを再確認

### 関連する原則

- **簡単にする**: 不要な手間を省き、必要な準備を明確にする
- **フィッツの法則**: 必要な情報に素早くアクセスできるようにする
- **ヒックの法則**: 選択肢を減らし、必要な行動を明確にする
- **タスクコヒーレンス**: 関連するタスクをまとめて提示する

### ビジネスへの影響

前提条件を後回しにすることで起こる問題：
- **離脱率の増加**: 途中で諦めるユーザーが増える
- **カスタマーサポートの負担増**: 「知らなかった」という問い合わせが増える
- **ブランドイメージの低下**: 「不親切」「時間の無駄」という印象を与える

前提条件を先に提示することで得られる利点：
- **完了率の向上**: 準備ができたユーザーだけが手続きを開始
- **ユーザー満足度の向上**: 透明性の高いプロセスに好印象
- **時間の節約**: ユーザーにとってもサービス提供者にとっても効率的
