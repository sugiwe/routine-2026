# ゼロ・ワン・インフィニティ (Zero One Infinity Rule)

- 日付: 2026-02-05
- URL: https://www.sociomedia.co.jp/9365
- daily 記録: [daily/2026-02-05.md](../../daily/2026-02-05.md)

## ユーザーの第一印象（daily より）

個数に関して恣意的な仕様を作らない。要素の存在可能な個数は基本的に0か1か無限とする。一覧の項目数、フォルダの階層数、入力欄の文字数など、ユーザーが増やせるものに数の制限をつけない。またどれだけ増えてもUIが破綻しないようにする。

## 深掘り・補足

### 核となる概念

ゼロ・ワン・インフィニティルールは、**恣意的な数値制限を避け、スケーラビリティを重視する設計原則**です。システムが扱う要素の個数は、基本的に以下の 3 パターンのみに限定すべきという考え方です：

- **0（ゼロ）**: 要素が存在しない
- **1（ワン）**: 要素が 1 つだけ存在する
- **∞（インフィニティ）**: 要素が無制限に存在できる

### なぜ恣意的な制限を避けるべきか

「最大 5 項目まで」「3 階層まで」といった制限は、多くの場合、技術的な必然性ではなく、設計者の都合や思い込みから生まれます。このような制限は：

- **ユーザーの創造性を阻害する**: 実際のニーズは設計者の想定を超える
- **システムの拡張性を損なう**: 後から制限を緩和するのは困難
- **一貫性のない体験を生む**: 「なぜここは 5 個まで、あっちは 10 個まで？」

### 良い例と悪い例

#### 悪い例

- **ツールバー**: 「アイコンは最大 8 個まで」と固定
  - 問題: ユーザーのニーズは多様。8 個では足りない人も、3 個で十分な人もいる

- **フォルダ階層**: 「3 階層までしかネストできません」
  - 問題: 複雑なプロジェクト管理では階層が深くなる必然性がある

- **入力欄**: 「コメントは 100 文字まで」
  - 問題: 詳細な説明が必要な場合、ユーザーは分割投稿を強いられる

#### 良い例

- **フォルダ構造**:
  - ルートフォルダ = 0 個の親フォルダ
  - サブフォルダ = 1 個の親フォルダ
  - 各フォルダ = 無限個の子要素を持てる

- **ツールバー**:
  - 表示領域に収まる分だけ表示
  - 溢れた分は「その他」メニューやスクロールで対応
  - ユーザーが自由にカスタマイズ可能

- **入力欄**:
  - 技術的上限（例: データベースの TEXT 型の上限）のみ設定
  - UI 上は自動拡張されるテキストエリアを提供

### UI が破綻しないための設計テクニック

無限に要素が増えても UI が破綻しないようにするための手法：

1. **スクロール**: 長いリストはスクロール可能にする
2. **ページネーション**: 大量データを分割表示
3. **仮想スクロール**: 表示領域の要素だけを描画（パフォーマンス最適化）
4. **検索・フィルタリング**: 大量データから目的の項目を見つけやすくする
5. **折りたたみ**: 階層構造を折りたたんで表示（ツリービュー）
6. **オーバーフローメニュー**: 表示しきれない項目は「…」メニューに格納

### 実践的なアドバイス

- **技術的制約は受け入れる**: データベースの上限、メモリ制約など、真の技術的制限は仕方ない
- **しかし、それを隠さない**: 制限がある場合は、その理由をユーザーに明示する
- **デフォルトは寛容に**: 「もっと少なくできる」より「もっと増やせる」方がユーザーフレンドリー
- **エッジケースをテストする**: 0 件、1 件、1000 件、10000 件でも UI が機能するか確認

### 関連する原則

- **Tesler's Law（複雑性保存の法則）**: 制限を設けても、ユーザーが回避策を見つけるだけで複雑性は減らない
