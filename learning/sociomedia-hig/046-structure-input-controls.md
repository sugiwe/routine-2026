# 46. 入力欄を構造化する (Structure Input Controls)

- 日付: 2026-02-15
- URL: https://www.sociomedia.co.jp/9470
- daily 記録: [daily/2026-02-15.md](../../daily/2026-02-15.md)

## ユーザーの第一印象（daily より）

これ悩ましいんよな、単純なテキストボックスでコピペして入れたい時もあるしカレンダーから選択できた方が楽な時もある

## 深掘り・補足

### 原則の核心

入力値の書式に合わせて入力欄を**分割**または**制限コントロール化**することで、以下を実現します：

1. **ユーザーに入力内容を示唆する** - 何を入力すべきかが明確になる
2. **入力操作を効率化する** - 適切な UI で入力が楽になる
3. **エラーを減らす** - 不正な値を入力できないようにする

**ただし注意**: 単純なテキストボックスと比べて操作効率が低下する場合もあるため、**情報の性質と利用シーン**を慎重に検討する必要があります。

### 4 つのアプローチ

#### 1. 単純なテキストボックス

```
[                    ]
```

- **メリット**:
  - キーボード習熟者の連続入力に最適
  - コピー & ペーストが自由
  - フレキシブル
- **デメリット**:
  - 認知負荷が高い（何を入力すべきか分かりにくい）
  - 入力ミスが起きやすい
  - バリデーションが事後的になる

#### 2. 分割したテキストボックス

```
[    ] / [    ] / [        ]
 MM      DD        YYYY
```

- **メリット**:
  - 書式が視覚的に示唆される
  - 認知負荷が低い
  - 入力すべき桁数が明確
- **デメリット**:
  - コピペがしにくい
  - フィールド間の移動が必要

#### 3. 制限コントロール（ドロップダウンなど）

```
[Month ▼] [Day ▼] [Year ▼]
```

- **メリット**:
  - 有効な値のみを入力させることができる
  - エラーが起きない
  - 選択肢が視覚化される
- **デメリット**:
  - 選択肢が多すぎると使いにくい（例: 年の選択）
  - キーボード操作の効率が落ちる

#### 4. 専用ピッカー（カレンダーなど）

```
[2026-02-15] [📅]
```

- **メリット**:
  - 視覚的にわかりやすい
  - 情報の性質に合わせた操作性（例: カレンダーで曜日も確認可能）
  - 楽しい UX
- **デメリット**:
  - 現在から離れた日付を指定する場合には意味がない（例: 生年月日）
  - モバイルでは画面を占有する
  - 実装コストが高い

### ユーザーの悩みへの回答

> 単純なテキストボックスでコピペして入れたい時もあるしカレンダーから選択できた方が楽な時もある

**これは正しい観察です！** 解決策は：

#### ハイブリッドアプローチ

多くの優れた UI は**両方を提供**します：

```
[2026-02-15        ] [📅]
 ↑ テキスト入力可    ↑ ピッカーを開く
```

- テキストボックスで直接入力・コピペ可能
- ピッカーボタンでビジュアルな選択も可能
- 入力後のバリデーションで整合性を確保

#### 利用シーンの考慮

| シーン | 推奨 UI | 理由 |
|--------|---------|------|
| 生年月日の入力 | 分割テキストボックス or ドロップダウン | 遠い過去の日付、カレンダーは非効率 |
| 予約日の選択 | カレンダーピッカー | 近い未来、曜日の確認も重要 |
| クレジットカード有効期限 | ドロップダウン（月/年） | 限定的な選択肢、ミス防止が重要 |
| 電話番号 | 分割テキストボックス | 書式の示唆、コピペも考慮 |
| データ一括入力（業務システム） | 単純テキストボックス | キーボード連続入力の効率を優先 |

### 関連する UI 原則

- **制限コントロール活用** - 不正な値を入力させない
- **値の入力より結果選択** - 選択肢から選ぶ方が楽
- **デフォルトボタンの具体性** - よく使われる値をデフォルトに
- **良いデフォルト** - 推測可能な値を事前設定

### 実践のポイント

1. **ユーザー層を見極める**
   - 一般ユーザー → ビジュアルな UI を優先
   - パワーユーザー → キーボード効率を優先

2. **利用頻度を考える**
   - 頻繁に入力 → 効率を優先
   - たまに入力 → わかりやすさを優先

3. **データの性質を理解する**
   - 定型的な値 → 制限コントロール
   - 自由形式 → テキストボックス

4. **両立を検討する**
   - ハイブリッド UI で柔軟性を確保
   - アクセシビリティも考慮

**結論**: 「これが正解」という単一解はありません。ユーザーのコンテクストとデータの性質を深く理解し、適切なバランスを見つけることが重要です。
