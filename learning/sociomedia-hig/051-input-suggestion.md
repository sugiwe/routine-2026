# 入力サジェスチョンを提示する (Input Suggestion)

- 日付: 2026-02-20
- URL: https://www.sociomedia.co.jp/9515
- daily 記録: [daily/2026-02-20.md](../../daily/2026-02-20.md)

## ユーザーの第一印象（daily より）

ユーザーは自分で入力するよりも選択肢から選ぶことを好むことが多いらしい

## 深掘り・補足

### 基本的な考え方

「タイピングは認知負荷が大きい」という事実がこの原則の出発点です。ユーザーが何かを入力しようとするとき、文字を思い出す・打ち間違えを修正する・変換する、といった複数のステップが発生します。サジェスチョンはこの負荷をまるごと引き受けてくれる仕組みです。

### サジェスチョンの 4 種類

| 種類 | 例 |
|------|----|
| **変換候補** | IME で「にほん」と入力すると「日本」「二本」などが表示される |
| **スペル補正** | 「recieve」と打つと「receive」を提案する |
| **検索キーワード** | 「UI デ」と打つと「UI デザイン」「UI デザイナー」などが候補に出る |
| **関連項目** | メールアドレス入力欄で、過去に使ったアドレスが候補として出る |

### なぜ「入力より選択を好む」のか

人間の行動モデルとして、**認識は想起より簡単**（Recognition over Recall）という原則があります（[012. Minimize the user's memory load](./012-minimize-memory-load.md) も参照）。
「〇〇という単語を思い出して入力する」より「提示された候補の中から正しいものを選ぶ」ほうが、脳への負荷が低いのです。

### 関連する原則との繋がり

- **[014. Precomputation](./014-precomputation.md)**: サジェスチョンは「ユーザーが必要とするであろう情報をあらかじめ計算・準備しておく」プリコンピュテーションの典型例
- **[013. Constraints](./013-constraints.md)**: 選択肢を限定することでユーザーの迷いを減らす
- **[045. 値を入力させるのではなく結果を選ばせる](./045-results-rather-than-values.md)**: サジェスチョンはこの原則の動的版とも言える

### 良い実装のポイント

- **タイミング**: 2〜3 文字入力された時点で候補を出す（早すぎると多すぎて役に立たない）
- **候補数**: 5〜10 件程度に絞る（多すぎると Hick's Law により選択コストが上がる）
- **順序**: 最も可能性が高いものを上位に（使用頻度・文脈・パーソナライズを活用）
- **キーボード操作対応**: マウスなしで候補を選べるよう、矢印キー＋Enter で確定できるようにする

### 悪い例

- 1 文字入力しただけで大量の候補が表示される → ノイズになり無視される
- 候補が関連性の低い順で並んでいる → ユーザーが探す手間が増える
- 候補を閉じる方法が分かりにくい → ユーザーが混乱する
