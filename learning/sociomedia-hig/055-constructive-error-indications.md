# 55. エラー表示は建設的にする (Constructive Error Indications)

- 日付: 2026-02-24
- URL: https://www.sociomedia.co.jp/9535
- daily 記録: [daily/2026-02-24.md](../../daily/2026-02-24.md)

## ユーザーの第一印象（daily より）

エラー発生時には、プログラマー向けのエラー番号とかではなく、わかりやすいメッセージによってユーザーの理解と行動を手助けする。

## 深掘り・補足

### 建設的なエラー表示の 3 要素

エラーメッセージは以下の 3 つの要素を含むべきです：

1. **何が起きたのか** - 状況の明確な説明
2. **なぜ起きたのか** - 原因の簡潔な説明（必要な場合）
3. **どうすればいいのか** - 具体的な解決策や次のアクション

### 悪い例と良い例

#### ❌ 悪い例

```
エラー: 0x80070057
致命的なエラーが発生しました
System Error: Invalid parameter
```

**問題点：**
- 技術的なエラーコードのみで一般ユーザーには意味不明
- 何が起きたのか、どうすればいいのかが不明
- プログラマー向けの専門用語

#### ✅ 良い例

```
ファイルを保存できませんでした
ディスクの空き容量が不足しています。
不要なファイルを削除してから、もう一度お試しください。
```

**優れている点：**
- 何が起きたのか明確（保存失敗）
- 原因が分かりやすい（容量不足）
- 具体的な解決策を提示（ファイル削除）

### 実践的なガイドライン

#### 言葉遣い

- **ユーザーの言語を使う** - 専門用語を避け、日常的な言葉で
- **肯定的な表現** - 「〜できません」だけでなく「〜してください」も含める
- **謝罪は適切に** - システムの不具合なら謝罪、ユーザーの入力ミスなら丁寧に説明

#### 情報の優先順位

1. **最優先**: 解決策（ユーザーが次に何をすべきか）
2. **重要**: 状況説明（何が起きたのか）
3. **補足**: 技術詳細（必要な場合のみ、別途表示）

#### エラーの深刻度に応じた設計

**軽微なエラー（修正可能）:**
```
メールアドレスの形式が正しくありません
「@」を含む有効なメールアドレスを入力してください
```

**重大なエラー（操作継続困難）:**
```
ネットワーク接続が切断されました
インターネット接続を確認してから、もう一度お試しください
サポートが必要な場合: support@example.com
```

### 技術情報の扱い

プログラマー向けの詳細情報が必要な場合：

- **「詳細を表示」ボタン** で折りたたんでおく
- **ログファイルへ記録** してサポート時に参照
- **エラーレポート機能** で自動送信（ユーザーの同意を得て）

### ユーザビリティへの影響

建設的なエラー表示は：

- ✅ ユーザーの不安を軽減
- ✅ 問題解決までの時間を短縮
- ✅ サポート問い合わせを減少
- ✅ システムへの信頼感を向上
