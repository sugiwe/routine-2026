# 56. 可能性と確率を区別する (Segregate Possible From Probable)

- 日付: 2026-02-25
- URL: https://www.sociomedia.co.jp/9540
- daily 記録: [daily/2026-02-25.md](../../daily/2026-02-25.md)

## ユーザーの第一印象（daily より）

「滅多にない事を考慮しすぎて通常の利用を面倒にしてはいけない。プログラマーはエッジケースを重視するがデザイナーはメインケースを重視すべき」とのこと。深い話だな👀 パレートの法則を思い出した。プログラマは、自動テストなどの活用で細かい検証を実施可能だからエッジケースを網羅すべきだが、少ないコストで最大のパフォーマスを出す、多くの人に誤解をさせない、マイノリティな要望は控えめに搭載してカバーする、みたいなことかな。

## 深掘り・補足

### 原則の核心

この原則は、**頻度と可能性を分けて考える**ことの重要性を説いています。技術的に「起こり得ること（Possible）」と、実際に「頻繁に起こること（Probable）」は全く別物であり、デザインでは後者を優先すべきです。

### プログラマーとデザイナーの視点の違い

- **プログラマー**：技術的に起こり得るあらゆるケース（エッジケース）を重視
  - バグを防ぐため、すべての可能性を検証
  - 自動テストで網羅的にチェック可能
- **デザイナー**：実際に頻繁に発生する通常使用（メインケース）を重視
  - ユーザーの大多数の行動パターンを優先
  - 稀なケースのために全体を複雑にしない

この視点の相違を理解し、バランスを取ることがインターフェース設計の鍵となります。

### 具体例：保存ダイアログ

**問題のある設計**：
「保存しますか？」ダイアログで「保存する」「保存しない」「キャンセル」を同等に扱う

**より良い設計**：
- **保存（確率高）をデフォルト**にする
- 「保存しない」は必要なユーザーのため、副次的な選択肢として配置
- さらに進めて、**自動保存をデフォルト化**することで誤消去リスクを削減

### 実践的なデザイン指針

1. **メインケースを最優先**
   - 80-90% のユーザーが行う操作を最も簡単にする
   - パレートの法則（80:20 ルール）を意識

2. **エッジケースは区別して配置**
   - オプション設定や詳細メニューに配置
   - メイン UI を複雑にしない
   - 必要な人だけがアクセスできるようにする

3. **段階的な開示（Progressive Disclosure）**
   - 基本機能はすぐ使える状態に
   - 高度な機能は「詳細設定」などに隠す

### 良い例と悪い例

**❌ 悪い例**：
- すべての設定項目を初期画面に表示
- 稀にしか使わない警告を毎回表示
- 「エキスパートユーザー」向けオプションを目立たせる

**✅ 良い例**：
- Gmail の「アーカイブ」ボタン（よく使う）vs「スパム報告」（稀）の配置
- iPhone の「設定」→「一般」→「詳細な設定」という階層構造
- Google 検索の「検索オプション」リンク（ほとんどのユーザーは使わない）

### パレートの法則との関連

ユーザーの印象通り、この原則はパレートの法則（80:20 ルール）と密接に関連しています：

- **80% のユーザー**は **20% の機能**しか使わない
- その 20% の機能を **最も使いやすく**設計すべき
- 残り 80% の機能は「あると便利」だが、**メインの使いやすさを犠牲にしてはいけない**

### プログラマーとしての実践

プログラマーは技術的にエッジケースを網羅すべきですが、**UI デザインでは異なるアプローチが必要**です：

- **コード**: エッジケースを徹底的にテストし、エラーハンドリングを実装
- **UI**: メインケースを最優先し、エッジケースは控えめに配置
- **ドキュメント**: エッジケースの対処法を記載（UI には出さない）

この「技術的厳密性」と「デザインの簡潔性」のバランスが、優れた製品を生み出します。
