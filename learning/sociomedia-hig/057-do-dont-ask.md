# 57. 黙って実行する (Do, Don't Ask)

- 日付: 2026-02-26
- URL: https://www.sociomedia.co.jp/9547
- daily 記録: [daily/2026-02-26.md](../../daily/2026-02-26.md)

## ユーザーの第一印象（daily より）

ユーザーがやろうとしたことにいちいち確認をとったり、処理が正常に完了したことを報告したりしない。UNIX哲学の「沈黙は金」に通じる話かな。

## 深掘り・補足

### 原則の定義

「黙って実行する」は、ユーザーの意図を尊重し、余計な確認ダイアログや完了報告を最小限にする UIX 原則です。ユーザーが操作を実行した時点で意思決定は完了しているため、システムは素早く実行すべきという考え方です。

### なぜ重要なのか

1. **ワークフローの流暢性**: 確認ダイアログはユーザーの思考と作業フローを中断させます
2. **効率性の向上**: 不要なクリックやキー操作を削減し、操作速度を向上
3. **信頼関係の構築**: システムがユーザーの意図を理解していることを示します
4. **認知負荷の軽減**: 「本当にやりたいのか？」と繰り返し問われることでの疲労を防ぎます

### 悪い例：過剰な確認ダイアログ

```
ファイル削除時:
「これらの2個の項目をごみ箱に移しますか？」
[キャンセル] [OK]
```

ユーザーは既に削除ボタンを押して意思表示をしているにもかかわらず、モーダルダイアログで作業を中断させるのは悪い設計です。

### 良い例：モードレスなフィードバック

- ファイルを削除したら、画面から消えてごみ箱アイコンに移動
- 「○個のファイルを削除しました」というトースト通知（数秒で消える）
- Undo 操作を可能にすることで安心感を提供

### 重要な例外：破壊的操作の場合のみ確認

**確認が必要な場合**:
- 不可逆的かつデータ消失の恐れがある操作
- 例: データベースの完全削除、アカウントの永久削除、大量データの一括削除

**確認が不要な場合**:
- 元に戻せる操作（Undo 可能）
- 復元可能な削除（ごみ箱に移動）
- 設定の変更（後で戻せる）

### フィードバックの設計原則

1. **画面の変化で伝える**: 状態の変化を視覚的に表現
2. **モードレスに**: ユーザーの作業を止めない形でフィードバック
3. **Undo を提供**: 確認の代わりに取り消し可能にする

### UNIX 哲学との共通点

UNIX の設計哲学「沈黙は金 (Silence is Golden)」と通じるものがあります：
- 正常に動作した場合は何も言わない
- エラーが発生した場合のみ報告する
- ユーザーを信頼し、余計な口出しをしない

### 実装時のベストプラクティス

1. **デフォルトは実行**: 確認を求めるのは例外的なケース
2. **Undo を優先**: 確認ダイアログよりも取り消し機能を充実させる
3. **進行状況は控えめに**: 長時間処理の場合のみプログレスバーを表示
4. **成功の沈黙**: 正常完了時の「成功しました！」ダイアログは不要
5. **エラーは明確に**: 問題が発生した場合のみ、具体的に伝える

### 関連するガイドライン

- **可能性と確率を区別する**: 本当に危険な操作だけ確認する
- **ガッツを見せる**: システムが自信を持って動作する
- **フェイルセーフ**: 確認よりも安全な設計で保護する
