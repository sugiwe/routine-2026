# 画面の変化をアニメーションで表す (Animate Screen Transitions)

- 日付: 2026-03-04
- URL: https://www.sociomedia.co.jp/9675
- daily 記録: [daily/2026-03-04.md](../../daily/2026-03-04.md)

## ユーザーの第一印象（daily より）

わかっちゃいるけどフロントエンド側を複雑にしたくないから最小限にしたいやつだ…

## 深掘り・補足

### 原則の核心

画面の広範な変化が発生する際に、トランジションのアニメーションを活用して状態遷移の連続性をユーザーが把握できるようにする、という原則です。

### なぜアニメーションが重要なのか

**1. ユーザーの認知的負荷を軽減**
- 突然の画面変化は、ユーザーに「今どこにいるのか」「何が起きたのか」という混乱を与える
- アニメーションは変化の「橋渡し」となり、状態遷移を理解しやすくする

**2. UIの応答性を向上**
- 瞬時の切り替えよりも、適度なアニメーション（0.1～0.5秒）の方が「スムーズで応答性が高い」と感じられる
- ユーザーは変化を追跡でき、システムが応答していることを実感できる

**3. 空間的な関係性を示す**
- どこから何が現れたのか、どこへ何が消えたのかを視覚的に示す
- メンタルモデルの構築を助ける

### 実装の具体例

**良い例:**
- モーダルダイアログが下からスライドインして現れる
- カードが拡大してディテールビューになる
- サイドバーがスライドして開閉する
- リストアイテムがフェードアウトして削除される

**悪い例:**
- ページ全体が突然切り替わる（何が起きたか追跡できない）
- 複数の要素が同時にバラバラに動く（混乱を招く）
- アニメーションが長すぎる（イライラする、効率が悪い）

### 技術的な考慮事項

**推奨される時間：0.1～0.5秒**
- 0.1秒未満: 変化を認識しづらい
- 0.5秒以上: ユーザーを待たせることになり、効率が悪い
- iOS は 0.3秒、Material Design は 200～300ms を標準としている

**パフォーマンス**
- CSS transitions/animations を使う（GPU アクセラレーション）
- JavaScript アニメーションは最小限に
- `transform` と `opacity` は高速、`width/height` は重い

**アクセシビリティ**
- `prefers-reduced-motion` メディアクエリに対応する
- モーションに敏感なユーザーのためにアニメーションを無効化できるようにする

### フロントエンド実装の現実的なバランス

ユーザーのメモにあるように「フロントエンド側を複雑にしたくない」という懸念は現実的です。以下のように優先度をつけると良いでしょう：

**必須（最小限の実装）:**
- モーダル・ドロワーの開閉
- ページ遷移（全画面の切り替え）
- 重要な状態変化（削除、追加など）

**望ましい（余裕があれば）:**
- ホバー効果
- リストのソート・フィルタリング
- 展開/折りたたみ

**過剰（避けるべき）:**
- すべての要素に細かいアニメーション
- 複雑なタイミング制御
- 不必要な装飾的アニメーション

### 関連ガイドライン

- **トランジションは両方向につける**: 開くときだけでなく閉じるときにも同じアニメーションを逆再生する
- **ダイレクトマニピュレーション**: ドラッグなど直接操作とアニメーションの組み合わせ

### まとめ

アニメーションは「見た目を良くする装飾」ではなく、**ユーザビリティを向上させる機能**です。しかし、実装コストとのバランスを考え、影響の大きい場所（画面全体の変化、重要なUI要素）に絞って適用することで、効率的に効果を得られます。
