# 74. ハイライト表現は構成要素をひとつだけ変化させる (Highlight By Changing Only One Element)

- 日付: 2026-03-15
- URL: https://www.sociomedia.co.jp/9770
- daily 記録: [daily/2026-03-15.md](../../daily/2026-03-15.md)

## ユーザーの第一印象（daily より）

例を見ると「そりゃそうだろ」って思うけど、実際のシーンで意外とやりがちなので気をつけたい

## 深掘り・補足

### コア原則

複数の同列項目の中でハイライト対象を示す際の基本ルール：

**「その項目の視覚的な構成要素をひとつだけ変化させる。あるいはひとつだけ追加する」**

### なぜこの原則が重要か

変化が多すぎると、ユーザーは「同じ種類の項目のハイライト」ではなく「別種類の要素」として認識してしまいます。ハイライトの目的は「注目させること」であって「分類を変えること」ではありません。

### 良い例

- **背景色のみを変える** - 最もシンプルで効果的
- **枠線色のみを変える** - 背景色が使えない場合に有効
- **枠線を追加する** - 項目数が少ない場合（2項目など）に適している

### 悪い例（避けるべきパターン）

- **形と背景色を同時に変える** - 変化が大きすぎて別の要素に見える
- **前景色と枠線色を同時に変える** - 複数の変化が混乱を招く
- **複数の装飾を追加する** - 「別のタイプ」として認識される

### 実践的なポイント

1. **変化の最小化を意識する**
   - 「目立たせたい」という気持ちから複数の要素を変えたくなるが、我慢する
   - 単一の変化でも十分に認識可能

2. **項目数に応じた手法を選ぶ**
   - 3項目以上：変化（色を変えるなど）
   - 2項目のみ：追加（枠線を加えるなど）

3. **一貫性を保つ**
   - 同じシステム内では同じハイライト手法を使う
   - ユーザーが学習した認識パターンを活用できる

### よくある失敗ケース

実務でよく見かける「やりすぎ」の例：

- 選択中のタブ：背景色 + 太字 + アイコン追加
- アクティブなボタン：色変更 + サイズ拡大 + 影追加
- フォーカス中の入力欄：枠線色 + 背景色 + ラベル色変更

これらは「目立つ」が「別の要素」に見えてしまうリスクがあります。

### デザインシステムへの応用

この原則をデザインシステムに組み込む際は：

- ハイライトのバリエーションを1つだけ定義する
- 「primary」「selected」「active」などの状態で同じ変化を適用
- レビュー時に「変化の数」をチェック項目に含める

シンプルさが、ユーザーの認知負荷を下げ、UIの質を高めます。
