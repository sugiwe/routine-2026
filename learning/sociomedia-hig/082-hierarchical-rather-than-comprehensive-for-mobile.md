# モバイルでは包括的より階層的に (Hierarchical Rather Than Comprehensive For Mobile)

- 日付: 2026-03-23
- URL: https://www.sociomedia.co.jp/9801
- daily 記録: [daily/2026-03-23.md](../../daily/2026-03-23.md)

## ユーザーの第一印象（daily より）

わかる。が、実際にこれを最適にやるのもむずくて、よく考えて設計したい

## 深掘り・補足

### 基本原則

モバイルとデスクトップの UI 設計には、根本的なアプローチの違いがあります。

- **デスクトップ**: 作業空間の全体像を包括的に現すのが良い
- **モバイル**: 一度に見せる情報を限定して階層的に見せるのが良い

### 具体例：iPhoto と Photos の比較

**Mac iPhoto（包括的設計）**

- アルバム、イベント、写真、操作ボタンが一画面に集約
- ユーザーがその場でほぼすべての操作が可能
- 大画面を活かして全体像を把握できる

**iPhone Photos（階層的設計）**

- アルバム一覧 → 写真一覧 → 写真表示がそれぞれ別画面
- ユーザーは階層を移動しながらアクセス
- 各画面の役割が明確で、認知負荷が低い

### 階層的設計のメリット

1. **小画面への最適化**: 限られたスペースを効率的に活用
2. **認知負荷の軽減**: 一度に処理する情報量を制限
3. **明確なフォーカス**: 各画面の役割が限定的でシンプル
4. **操作の明確化**: ユーザーが次に何をすべきか分かりやすい

### 階層的設計の課題

1. **階層の深さ**: 深すぎると目的の画面にたどり着くのが困難
2. **方向感覚の喪失**: 現在位置が分からなくなるリスク
3. **操作の増加**: 画面遷移の回数が増える
4. **コンテキストの喪失**: 全体像が見えにくくなる

### 実践的なポイント

#### 1. 階層の最適化

- **3階層ルール**: 可能な限り3階層以内に収める
- **フラット化**: 不要な階層は省く
- **ショートカット**: よく使う機能への直接アクセスを提供

#### 2. ナビゲーションの明確化

- **パンくずリスト**: 現在位置を明示
- **戻るボタン**: 一つ前の階層への移動を簡単に
- **タイトルバー**: 現在の画面の役割を明確に表示

#### 3. トレードオフの判断

階層を増やすべきかどうかの判断基準：

- **増やすべき**: スクリーン内の認知負荷が高すぎる場合
- **避けるべき**: 頻繁に行き来する必要がある操作の場合

#### 4. 良い例と悪い例

**良い例**

- 設定アプリ: カテゴリー → 詳細設定 → 個別項目
- EC サイト: カテゴリー → 商品一覧 → 商品詳細

**悪い例**

- 5階層以上の深い構造
- 関連する情報が別々の階層に散在
- 頻繁に使う機能が深い階層に埋もれている

### 設計時の考慮事項

ユーザーの第一印象にある「実際にこれを最適にやるのもむずくて、よく考えて設計したい」という点は非常に重要です。

階層的設計を成功させるには：

1. **ユーザーの行動パターンを分析**: よく使う機能は浅い階層に
2. **情報のグルーピング**: 関連する情報をまとめる
3. **プロトタイプでテスト**: 実際のユーザーでナビゲーションの快適さを検証
4. **データで検証**: 階層ごとの離脱率や操作時間を測定

### まとめ

モバイル UI における階層的設計は、小画面の制約を逆手に取った効果的な戦略です。ただし、「階層を増やせば良い」というわけではなく、ユーザーの目的達成を最も簡単にする構造を慎重に設計する必要があります。包括的設計と階層的設計のバランスを取りながら、最適な情報アーキテクチャを構築することが重要です。
