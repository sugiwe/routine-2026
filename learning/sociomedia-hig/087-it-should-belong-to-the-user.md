# ユーザーの道具にする (It Should Belong To The User, Not The Provider)

- 日付: 2026-03-28
- URL: https://www.sociomedia.co.jp/9824
- daily 記録: [daily/2026-03-28.md](../../daily/2026-03-28.md)

## ユーザーの第一印象（daily より）

某通販サイトとかでこれに反しているのをよく見かける気がする😅

## 深掘り・補足

### 核心的な考え方

この原則は、システム設計における **所有権の概念を根本的に転換** するものです。"It Should Belong To The User, Not The Provider" という英語表現が示すように、サービスはプロバイダー（提供者）のものではなく、**ユーザーが主体的に利用するための道具** として設計すべきだという考え方です。

提供者の営業目的を優先するのではなく、ユーザーが自身の活動を強化するための手段として位置づけることが重要です。

### 具体例：EC サイトの設計思想

#### 悪い例：提供者視点の設計
- EC サイトを「売るための仕組み」として設計
- 不要な新製品広告をデフォルトでオンにする
- メールマガジンを自動的に購読させる
- おすすめ商品を過度にプッシュする

#### 良い例：ユーザー視点の設計
- EC サイトを「買うための仕組み」として設計
- ユーザーの購買活動を支援する
- 必要な情報だけを表示する
- ユーザーが主体的に選択できるようにする

提供者視点では営業活動が自然に思えても、ユーザー視点では「不要である可能性が高いオプションがデフォルトでオンになっている」ことは不自然です。

### よくある問題パターン

1. **デフォルトで有効な不要オプション**
   - メールマガジンの自動購読
   - クッキー同意の曖昧な選択肢
   - 個人情報の第三者提供への同意

2. **プロバイダーの都合を優先した UI**
   - 解約ボタンが見つけにくい
   - 退会手順が複雑
   - プレミアムプランへの誘導が過度

3. **ユーザーの選択を尊重しない**
   - 設定を勝手にリセット
   - 通知をオフにしても再度オンになる
   - ダークパターンによる誘導

### 実践のポイント

1. **ユーザーの意図を尊重する**
   - デフォルト設定はユーザーにとって最も安全で控えめなものにする
   - オプトイン方式を採用する（オプトアウトではなく）

2. **透明性を確保する**
   - 何が起こるかを明確に説明する
   - 選択肢を明示的に提示する

3. **ユーザーにコントロールを与える**
   - いつでも設定を変更できるようにする
   - 簡単に解除・退会できるようにする

### 関連する原則

- **ユーザーコントロール**: ユーザーが常に主導権を持つべき
- **直接操作**: ユーザーが直接的に操作できるようにする
- **モードレス設計**: ユーザーのペースで作業できるようにする
- **透明性**: システムの動作を明確にする

### ユーザー体験への影響

この原則に従うことで：
- **信頼感の向上**: ユーザーはサービスを信頼しやすくなる
- **満足度の向上**: ユーザーは自分のペースで利用できる
- **長期的な関係構築**: 短期的な利益よりも長期的な信頼関係を築ける

逆に、この原則に反すると：
- **ユーザーの離反**: ダークパターンに気づいたユーザーは離れていく
- **ブランドイメージの低下**: 評判が悪化する
- **法的リスク**: 消費者保護法に抵触する可能性

### まとめ

「ユーザーの道具にする」とは、ユーザーを主役にした設計思想です。提供者の都合ではなく、ユーザーの目的達成を第一に考えることで、より良いユーザー体験と長期的な信頼関係を築くことができます。
