# 089. ユーザーを教育するのではなくユーザーが学習できるようにする (Let Users Learn, Don't Educate Them)

- 日付: 2026-03-30
- URL: https://www.sociomedia.co.jp/9852
- daily 記録: [daily/2026-03-30.md](../../daily/2026-03-30.md)

## ユーザーの第一印象（daily より）

これも「話としちゃわかるけど、実践で気づいて最適な形にするのはむずい」やつだ。そういうものばかりだな、結局は良い例・悪い例を実際にたくさん見ながら経験を積んでいくのが重要そう

## 深掘り・補足

### 主要概念：自己説明的な設計

このガイドラインは、UI設計において「説明文に依存する」のではなく、「システムそのものが直感的に理解できる」ように設計すべきという原則です。

**重要なのは**：
- ユーザーに操作方法を「教育する」のではなく
- ユーザーが自力で「学習できる」環境を作ること

### なぜ重要なのか

**説明文の問題点**：
1. **読まれない**: ユーザーは長い説明文を読まずに、まず触ってみる
2. **認知負荷**: 説明を読んで理解する労力がかかる
3. **保守コスト**: 機能が変わるたびに説明文も更新が必要

**自己説明的設計の利点**：
1. **直感的**: 対象領域の知識があれば、見ただけで理解できる
2. **効率的**: 説明を読む時間が不要
3. **発見しやすい**: 視覚的手がかりで機能が自明になる

### 良い例と悪い例

#### 悪い例：説明に依存

```
「売上レポートをダウンロードするにはこちらをクリックしてください」
```

- ユーザーは説明文を読まないと何ができるかわからない
- 「こちら」というリンクだけでは意味が不明
- 説明文がないと機能を発見できない

#### 良い例：自己説明的

```
[ 売上レポートをダウンロード 🔽 ]
```

- ボタンのラベルが動詞で始まり、何ができるか明確
- アイコンがダウンロード機能を視覚的に示唆
- 説明文がなくても機能が自明

### 実践のポイント

#### 1. シグニファイアを活用する

**視覚的手がかりで機能を示す**：
- ボタンは押せそうな見た目にする
- リンクは下線やカラーで識別可能にする
- ドラッグ可能な要素にはハンドルを表示

#### 2. メンタルモデルに合わせる

**ユーザーの期待に沿った動作**：
- ゴミ箱アイコンなら削除機能
- フォルダアイコンなら整理・保存機能
- 既存の慣習に従う

#### 3. 一貫性を保つ

**同じ操作は同じ結果を**：
- 同じアイコンは常に同じ機能
- 同じジェスチャは常に同じ動作
- プラットフォームの慣習に従う

#### 4. 直接操作を可能にする

**仲介者を減らす**：
- ドラッグ&ドロップで直接移動
- インライン編集で直接修正
- リアルタイムプレビューで結果を即座に表示

### ユーザーの洞察：経験の重要性

ユーザーの第一印象「良い例・悪い例を実際にたくさん見ながら経験を積んでいくのが重要」は、まさに核心を突いています。

**デザイナー自身も「学習」が必要**：
- 理論を知るだけでは不十分
- 実際のUIを評価する経験が必要
- 良い例・悪い例のパターンを蓄積する
- ユーザビリティテストで検証する

このガイドライン自体が、デザイナーに対して「教育するのではなく学習を促す」アプローチを取っているとも言えます。
